# Task 3.2 - Authentication Design 
=======================================================

## a. Two-Factor Authentication (2FA) Proposal

**Possession-based factor:** A Time-based One-Time Password (TOTP) generated by an authenticator app such as Google Authenticator or Microsoft Authenticator.

**How it works:** After entering the username and password, the student is asked to enter a 6-digit code generated by the authenticator app. The app and the SARS server share a secret key, allowing both to generate the same code every 30 seconds. Access is granted only if the entered code matches the server's generated code.

**Why it improves security:** Even if an attacker captures the student's username and password, they still cannot log in without the registered mobile device that generates the one-time code. Since the code changes every 30 seconds, it cannot be reused, making it much harder to impersonate a user.

## b. Principle of Least Privilege for the Administrator
The Principle of Least Privilege ensures an account holds only the minimum permissions required for its role.

Students Table: READ only.

Enrollments Table: READ and WRITE.

Financial Records: Explicitly Denied at the database/role level (e.g., via REVOKE). Relying solely on UI-layer restrictions is insufficient, as database layer access could still be bypassed.

**Benefit if this account is compromised:** If these credentials are stolen, the blast radius is tightly contained. The attacker is restricted to viewing student data and modifying enrollments; they cannot access sensitive financial records because those permissions were never granted at the database level.

## c. Cryptographic Primitive for Integrity Checking
Most appropriate primitive: Hashing.
Why the others are not suitable for this specific purpose:

**Symmetric Encryption:** Built for confidentiality, not integrity. It does not generate a fixed-size fingerprint for easy comparison. Furthermore, encrypting an entire record just to detect changes introduces unnecessary computational overhead and conflates data hiding with data verification.

**Asymmetric Encryption:** Designed for confidentiality or digital signatures (authentication/non-repudiation). It is computationally far more expensive than hashing and is inefficient for a simple "has this data changed?" check where proving authorship is not required.

***Reason:***

*The Mechanism:* A cryptographic hash function (like SHA-256) converts any input into a fixed-size digest; due to collision resistance, even a single-character change generates a completely different digest, making it ideal for alteration detection without requiring secrecy.

*Storage Phase:* When a marks record is written, SARS computes SHA-256(record_contents) and stores this unique hash alongside the data.

*Verification Phase:* Upon retrieval, SARS recomputes the current record's hash and compares it directly to the originally stored hash.

*Tamper Detection:* A perfect match confirms data integrity, while any mismatch instantly reveals that the record was tampered with between storage and display.

## d. Non-Repudiation
**Non-repudiation** ensures that a person cannot deny performing an action, such as approving or signing a student's marks record. Unlike **integrity**, which only confirms that the data has not been changed, non-repudiation also proves who created or approved the data.
Non-repudiation requires **asymmetric cryptography** because only the owner of the **private key** can create a digital signature, while anyone with the corresponding **public key** can verify it. This proves the identity of the signer.

In the SARS system:

**Private signing key:** Held by the registrar or authorized faculty member who finalizes student marks.
**Public verification key:** Used by the SARS application, students, or auditors to verify the digital signature.

### d. Why RSA Security Relies on the Difficulty of Factoring n

The Core Dependency: Deriving the private exponent $d$ requires $\phi(n)$, which can only be calculated if the prime factors $p$ and $q$ of $n$ are known.The Security Mechanism: While factoring a small toy number like $n = 55$ is trivial, factoring a product of two several-hundred-digit primes is computationally impossible for classical algorithms in a feasible timeframe.The Conclusion: Because an attacker cannot practically factor $n$ at scale, they cannot compute $\phi(n)$ or the modular inverse $d$, preventing the public key $(e, n)$ from leaking the private key.

### d. Why the Attacker Can't Compute K, and the Vulnerability Without Authentication

The Math (Discrete Logarithm Problem): Intercepting public parameters ($p=23, \alpha=5, A=8, B=19$) is useless without the private exponents ($a$ or $b$). # Task 3.5 — TLS, Firewalls, and OS Hardening 
===================================================

### a. The TLS Handshake (Four Main Steps)

Step 1 — ClientHello: The browser sends its supported TLS versions, a list of compatible cipher suites, a "client random" number, and a key-share proposal. No secret data is exposed.

Step 2 — ServerHello & Certificate: The server selects a cipher suite, sends its own "server random", provides its CA-signed digital certificate (containing its public key), and shares its key-exchange material.

Step 3 — Verification & Key Derivation: The browser validates the server's certificate. Both sides then combine the random numbers and key-share material to independently compute the same symmetric session key.

Step 4 — Finished Messages: Both parties exchange an encrypted "Finished" message to verify that the entire handshake was received intact and that the keys match.

### b. Firewall vs. IDS

Attack the Firewall Prevents: By restricting incoming traffic to ports 80 and 443, the firewall blocks direct internet-facing connection attempts to unexposed services—such as an attacker trying to brute-force SSH (port 22) or exploit a database port (e.g., 5432/3306). It drops this traffic before it ever reaches the server.

Attack the IDS Detects (Firewall Misses): A SQL injection attack (e.g., ' OR '1'='1) sent inside a legitimate HTTPS request to port 443. Because the firewall only filters traffic by IP and port, it cannot inspect application-layer payloads. An IDS performs deep packet inspection to recognize the malicious SQL pattern and alert on the threat.

### c. OS Hardening Steps for the New Linux Server

| Step | Security Measure | Description |
|------|------------------|-------------|
| **1** | **Apply OS and Security Updates** | Run `sudo apt update && sudo apt upgrade -y` (or `yum update -y` on RHEL-based systems) after server setup and enable automatic security updates to keep the system protected from known vulnerabilities. |
| **2** | **Disable Unnecessary Services** | Stop and disable services that are not required, such as Telnet or FTP, to reduce the attack surface and prevent unnecessary network exposure. |
| **3** | **Disable Direct Root Login** | Set `PermitRootLogin no` in `/etc/ssh/sshd_config` and require administrators to use individual accounts with `sudo`. This improves accountability and reduces security risks. |
| **4** | **Use SSH Key-Based Authentication** | Set `PasswordAuthentication no` in `/etc/ssh/sshd_config` and allow only SSH key authentication. This protects the server from password brute-force attacks. |
